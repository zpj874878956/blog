## VictoriaMetrics 架构概览

以下是 VictoriaMetrics 的核心组件架构图:

![image-20260525092246454](https://pdf.chatquanttail.top/2026/05/20260525092246559.png)

- `vmstorage` 负责存储数据，是有状态组件。
- `vmselect` 负责查询数据，Grafana 添加 Prometheus 数据源时使用 `vmselect` 地址，查询数据时，`vmselect` 会调用各个 `vmstorage` 的接口完成数据的查询。
- `vminsert` 负责写入数据，采集器将采集到的数据 "吐到" `vminsert`，然后 `vminsert` 会调用各个 `vmstorage` 的接口完成数据的写入。
- 各个组件都可以水平伸缩，但不支持自动伸缩，因为伸缩需要修改启动参数。

## 安装 operator

使用 helm 安装:

```bash
helm repo add vm https://victoriametrics.github.io/helm-charts
helm repo update
helm install victoria-operator vm/victoria-metrics-operator

#指定参数安装
  helm install vm ./victoria-metrics-operator-0.63.1.tgz \
  --namespace monitoring \
  --timeout 15m \
  --set vlstorage.persistentVolume.storageClassName=nfs-monitor \
  --set "vlstorage.retentionPeriod=180d" \
  --set "vlstorage.persistentVolume.size=1Gi" \
  --set vmauth.enabled=true \
  --set vlstorage.replicaCount=2 \
  --set global.image.registry=docker.1ms.run \
  --set vlstorage.image.registry=docker.1ms.run \
  --set vlinsert.image.registry=docker.1ms.run \
  --set vlselect.image.registry=docker.1ms.run
```

检查 operator 是否成功启动:

```bash
$ kubectl -n monitoring get pod
NAME                                                           READY   STATUS    RESTARTS   AGE
victoria-operator-victoria-metrics-operator-7b886f85bb-jf6ng   1/1     Running   0          20s
```

## 安装 VMSorage, VMSelect 与 VMInsert

准备 `vmcluster.yaml`:

```bash
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMCluster
metadata:
  name: vmcluster
  namespace: monitoring
spec:
  retentionPeriod: "1" # 默认单位是月，参考 https://docs.victoriametrics.com/Single-server-VictoriaMetrics.html#retention
  vmstorage:
    replicaCount: 2
    storage:
      volumeClaimTemplate:
        metadata:
          name: data
        spec:
          accessModes: [ "ReadWriteOnce" ]
          storageClassName: cbs
          resources:
            requests:
              storage: 100Gi
  vmselect:
    replicaCount: 2
  vminsert:
    replicaCount: 2
```

安装:

```bash
$ kubectl apply -f vmcluster.yaml
vmcluster.operator.victoriametrics.com/vmcluster created
```

      authorization:

      - role: node
      relabel_configs:
      - source_labels: [__meta_kubernetes_node_label_node_kubernetes_io_instance_type]

        regex: __meta_kubernetes_node_label_(.+)
    - job_name: "tke-probes"
      scheme: https
      metrics_path: /metrics/probes
      tls_config:
        insecure_skip_verify: true

        separator: ;
        regex: (.*)

        replacement: ${1}
        action: replace

# 配置告警

**使用 VictoriaMetrics Operator 安装时，添加 vmalert 告警规则的正确方式是使用 `VMRule` CRD，而不是直接修改配置文件。**

Operator 会自动根据 `VMRule` 生成 ConfigMap 并挂载到 vmalert Pod 中，修改后会自动 reload，无需重启。

### 1. 在你的 VMAlert 中启用规则选择

确保你的 `VMAlert` CR 有正确的 selector（推荐）：

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMAlert
metadata:
  name: vmalert-prod
  namespace: monitoring
spec:
  # ... 其他配置 (datasource, notifiers 等)
  
  # 推荐：选择所有 namespace 的规则（或根据需要调整）
  ruleNamespaceSelector: {}     # 空 = 所有 namespace
  ruleSelector: {}              # 空 = 匹配所有 VMRule，或加 label 做过滤
  
  # 或者只选择特定 label 的规则
  # ruleSelector:
  #   matchLabels:
  #     app: production
```

### 2. 创建 VMRule 添加告警规则

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMRule
metadata:
  name: my-custom-alerts
  namespace: monitoring          # 可以和 VMAlert 在不同 namespace
  labels:
    app: production              # 如果 VMAlert 用 ruleSelector 过滤，这里要匹配
spec:
  groups:
    - name: my-app-rules         # 组名，建议按业务/系统分组
      interval: 30s              # 可选，评估间隔
      rules:
        # ==================== 告警规则示例 ====================
        - alert: HighCPUUsage
          expr: |
            sum(rate(node_cpu_seconds_total{mode!="idle"}[5m])) by (instance) > 0.8
          for: 5m
          labels:
            severity: warning
            team: sre
          annotations:
            summary: "Instance {{ $labels.instance }} CPU usage high"
            description: "CPU usage is {{ $value | humanizePercentage }} for more than 5 minutes."

        - alert: HighMemoryUsage
          expr: |
            (node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes) / node_memory_MemTotal_bytes > 0.85
          for: 3m
          labels:
            severity: critical
          annotations:
            summary: "High memory usage on {{ $labels.instance }}"

        # ==================== 记录规则示例 ====================
        - record: job:node_cpu:rate5m
          expr: sum(rate(node_cpu_seconds_total[5m])) by (job, instance)
```

**应用方式**：

```bash
kubectl apply -f my-rules.yaml
```

### 3. 验证是否生效

```bash
# 查看生成的 ConfigMap（Operator 会自动创建多个）
kubectl get configmap -l app.kubernetes.io/name=vmalert -n monitoring

# 查看 vmalert 日志
kubectl logs -l app.kubernetes.io/name=vmalert -n monitoring --tail=100

# 进入 vmalert 查看规则
kubectl port-forward svc/vmalert-prod 8880:8880
# 浏览器访问 http://localhost:8880/api/v1/rules
```

### 最佳实践建议

1. **按业务拆分**：一个业务/系统一个 `VMRule`，便于维护。
2. **使用 labels** 做区分：给 `VMRule` 加 `environment: prod`、`team: backend` 等标签，然后在 `VMAlert` 用 `ruleSelector` 精确匹配。
3. **记录规则**（recording rules）建议单独一个 group，提高性能。
4. **模板**：`annotations` 支持 Prometheus 模板语法（如 `{{ $value }}`、`{{ $labels.pod }}`）。

# 配置邮箱告警

**✅ 配置邮箱告警通知的正确方式（VictoriaMetrics Operator）**

VictoriaMetrics 的 `vmalert` **本身不直接发邮件**，它只负责评估规则并把告警发送给 **Alertmanager**（通过 notifier）。  
真正的邮件发送由 **VMAlertmanager** 负责。

### 1. 部署 VMAlertmanager（如果还没部署）

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMAlertmanager
metadata:
  name: vmalertmanager
  namespace: monitoring
spec:
  replicaCount: 1
  # 可选：资源限制等
```

### 2. 创建 VMAlertmanagerConfig 配置邮箱（推荐方式）

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMAlertmanagerConfig
metadata:
  name: email-config
  namespace: monitoring
spec:
  route:
    receiver: email   # 默认接收器
    group_by: ['alertname']
    group_wait: 30s
    group_interval: 5m
    repeat_interval: 4h

  receivers:
    - name: email
      email_configs:
        - to: 'your-team@example.com'           # 收件人（支持多个，用逗号分隔）
          from: 'alert@victoriametrics.example.com'  # 发件人
          smarthost: 'smtp.example.com:587'     # SMTP 服务器地址:端口
          hello: 'victoriametrics.example.com'
          auth_username: 'your-smtp-username'    # 如果需要认证
          auth_password:
            name: smtp-secret                    # Secret 引用（推荐）
            key: password
          auth_identity: 'your-smtp-username'    # 可选
          require_tls: true                      # 大部分邮箱需要
          send_resolved: true                    # 恢复时也发邮件
          headers:
            Subject: '[{{ .Status | toUpper }}] {{ .CommonLabels.alertname }}'

  # 可选：按 severity 路由
  # routes:
  #   - receiver: email
  #     matchers:
  #       - severity =~ "critical|warning"
```

### 3. 创建 SMTP 密码 Secret（强烈推荐，不要明文写密码）

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: smtp-secret
  namespace: monitoring
type: Opaque
data:
  password: <base64 编码后的密码>   # echo -n 'yourpassword' | base64
```

### 4. 在 VMAlert 中指向 VMAlertmanager

确保你的 `VMAlert` CR 有以下配置：

```yaml
apiVersion: operator.victoriametrics.com/v1beta1
kind: VMAlert
metadata:
  name: vmalert
  namespace: monitoring
spec:
  datasource:
    url: "http://your-vmselect-or-vmsingle:8428"
  
  notifiers:
    - url: "http://vmalertmanager.monitoring.svc.cluster.local:9093"
  
  # 其他配置...
```

### 5. 应用并验证

```bash
kubectl apply -f smtp-secret.yaml
kubectl apply -f vmalertmanagerconfig.yaml
kubectl apply -f vmalert.yaml   # 如果修改了
```

**验证命令**：

```bash
# 查看生成的 Alertmanager 配置
kubectl get secret vmalertmanager-vmalertmanager-generated -n monitoring -o yaml | grep alertmanager.yaml -A 50

# 查看 VMAlertmanager 日志
kubectl logs -l app.kubernetes.io/name=vmalertmanager -n monitoring
```

### 常见 SMTP 配置示例

- **腾讯企业邮**：`smarthost: smtp.exmail.qq.com:587`
- **阿里云邮箱**：`smarthost: smtp.aliyun.com:465`（SSL）
- **Gmail**：`smarthost: smtp.gmail.com:587`（需开启“应用专用密码”）
- **Outlook/Office365**：`smarthost: smtp.office365.com:587`
